# Mycellium — Apple (iOS + macOS) client (#68 + #69)

A thin SwiftUI UI over the `mycellium-sdk` UniFFI **Swift** bindings. No protocol,
crypto, storage, or network logic lives here — the app renders SDK state and
forwards user intent; everything else is behind the SDK boundary. This is the
Apple implementation of the shared blueprint in
[`docs/NATIVE-CLIENTS.md`](../../docs/NATIVE-CLIENTS.md); read that first. Nothing
here modifies any Rust crate.

This directory has **two** consumers of one generated Swift binding:

1. A **SwiftPM package** (`Package.swift`) that builds + tests the
   SDK-integration *core* — the generated binding, a thin facade, the secret
   stores, and a **real messaging round-trip XCTest**. It links the **host**
   `libmycellium_sdk.so`, so it builds and tests on **Linux/CI, no Mac required**.
2. A **SwiftUI app** (`App/`) built with **Xcode on a Mac**, linking an Apple
   `xcframework`. It is deliberately **excluded** from the SwiftPM targets so
   `swift build` on Linux never tries to compile SwiftUI/UIKit/Security.

> **Verified on Linux.** With Swift 6.3.2 + Rust installed, `./build-rust.sh`
> then `swift build` and `swift test` are green here: the `mycellium-sdk`
> compiles to a host `.so`, the UniFFI Swift binding generates, this package
> compiles against it, and the messaging round-trip test passes end-to-end
> against a live dev directory + queue. The SwiftUI app under `App/` is Mac-only
> and is not built here.

## Linux path (verified) — build + test the core

Prerequisites: **Rust** (`cargo`) and **Swift** (`swift --version` ≥ 5.9; verified
with 6.3.2).

```sh
cd clients/apple

# 1. Build the host libmycellium_sdk.so AND generate the Swift binding + C header.
#    Outputs (all gitignored build artifacts):
#      ../../target/debug/libmycellium_sdk.so
#      Sources/MyceliumSDK/Generated/mycellium_sdk.swift
#      Sources/mycellium_sdkFFI/mycellium_sdkFFI.h
./build-rust.sh

# 2. Compile the SDK-integration core.
swift build

# 3. Start a dev directory + queue for the round-trip test (another shell).
#    MYCELLIUM_DEV_AUTH=1 makes the directory echo the verification code back
#    (dev_code), so the flow works without a real inbox.
( cd ../.. && cargo build --release -p mycellium-server -p mycellium-queue )
MYCELLIUM_DEV_AUTH=1 ../../target/release/mycellium-server --addr 127.0.0.1:19080 &
../../target/release/mycellium-queue --addr 127.0.0.1:19090 &

# 4. Run the messaging round-trip test.
MYCELLIUM_DIR_URL=http://127.0.0.1:19080 \
MYCELLIUM_QUEUE_URL=http://127.0.0.1:19090 \
  swift test
```

The round-trip test (`MessagingRoundTripTests`) creates two `MyceliumClient`s in
temp dirs, runs each through `startEmailVerification` → `confirmEmailVerification`
→ `register`, sends alice → bob with `sendText`, polls `bob.sync()` until the
message decrypts, and asserts it lands in `bob.thread(alice)` and the
conversation list — plus a listener test and a `FileSecretStore` contract test.
Endpoints default to `127.0.0.1:19080` / `:19090`; override with
`MYCELLIUM_DIR_URL` / `MYCELLIUM_QUEUE_URL`. If the servers aren't reachable the
server-backed tests **skip** (so `swift test` stays friendly locally); CI starts
them and expects the round-trip to run.

> **Runtime linking.** The package embeds an rpath to the `.so` dir, so the test
> binary finds `libmycellium_sdk.so` without `LD_LIBRARY_PATH`. If you relocate
> the lib, set `MYCELLIUM_RUST_LIB_DIR` (or `LD_LIBRARY_PATH=../../target/debug`).
>
> **Toolchain note (Linux).** Some Swift-for-Linux packages ship `libIndexStore.so`
> outside the toolchain's own `lib/`. If `swift test` fails with
> `libIndexStore.so: cannot open shared object file`, symlink the system copy in,
> e.g. `ln -s /usr/lib64/libIndexStore.so <swift>/lib/libIndexStore.so`.

## Mac / Xcode path — build + run the SwiftUI app

The SwiftUI app needs **Xcode / macOS**; it will not build on Linux (SwiftUI /
UIKit / Security). One generated Swift binding is reused; only the native link
target differs (an Apple `xcframework` instead of the host `.so`).

```sh
cd clients/apple

# 1. Add the Apple Rust targets (once).
rustup target add aarch64-apple-ios aarch64-apple-ios-sim x86_64-apple-ios \
                  aarch64-apple-darwin x86_64-apple-darwin

# 2. Generate the Swift binding + header (same artifacts the Linux path uses).
./build-rust.sh

# 3. Build the per-arch static libs and bundle them into an xcframework.
#    (See the header comment in build-rust.sh for the exact cargo/lipo/
#    xcodebuild -create-xcframework invocation -> MyceliumFFI.xcframework.)

# 4. Generate + open the Xcode project.
brew install xcodegen            # once
xcodegen generate --spec App/project.yml
open Mycellium.xcodeproj
```

The app compiles `App/` plus the shared `Sources/MyceliumSDK` +
`Sources/MyceliumSecrets` sources and links `MyceliumFFI.xcframework`. In Xcode,
add your signing team and (for #71) the **Push Notifications** capability; the
SDK's `registerPush(platform: .apns(topic:), token:)` addresses the wake to the
app's bundle id.

### App flows (each backed by real SDK calls, mirroring the Android app)

- **Setup** — directory + queue URLs, persisted via `setSetting`.
- **Onboarding** — handle + email → `startEmailVerification` → enter the code →
  `confirmEmailVerification` → `register`.
- **Conversations** — `conversations()`.
- **Thread** — `thread(peer)` transcript + compose → `sendText`, with the
  optimistic `DeliveryState` shown per sent message.
- **Contacts / verify** — `addContact`, `contacts()`, `safetyNumber` to compare
  out of band, `markVerified`.
- Foreground receive via `sync()` on `.active` + a light poll, plus a registered
  `EventListener` so inbound messages (`onMessage`) surface live.

All SDK methods **block**; the view-model runs each off the main actor and maps
`SdkError` variants to user-facing messages.

## Secure storage — how this satisfies #65 on Apple

The app **never** uses the plaintext dev constructor `MyceliumClient(dataDir:)`.
`ClientHolder` builds the client with the production constructor:

```swift
try MyceliumClient.newWithSecretStore(dataDir, secrets: KeychainSecretStore(namespace: ""))
```

`KeychainSecretStore` (the #65 Apple adapter, in `Sources/MyceliumSecrets`)
implements the SDK's `SecretStore` seam with the **iOS/macOS Keychain**
(`Security` framework):

- each secret the SDK stores (today the ~64-byte identity under `"identity"`) is
  a `kSecClassGenericPassword` item added/updated via `SecItemAdd` /
  `SecItemUpdate`, read via `SecItemCopyMatching`, removed via `SecItemDelete`;
- the item is `kSecAttrAccessibleWhenUnlockedThisDeviceOnly` — hardware-protected
  while locked, and **excluded from iCloud Keychain / device-transfer backups**,
  so the account root key never leaves the device via restore (mirrors Android's
  `allowBackup=false`);
- it is **namespaced** per account: the Keychain `service` carries a namespace
  suffix (`eu.mycellium.identity[.<namespace>]`), and the item is keyed by
  `(service, account=key)`. A single-account app uses an empty namespace; an app
  holding more than one account passes a distinct namespace per account — so two
  accounts' `"identity"` items never overwrite each other. (This is the Android
  bug — a single store dir made two accounts collide — avoided by construction.)
- it **fails closed**: `load` returns `nil` **only** for `errSecItemNotFound`; any
  other Keychain error throws `SdkError.Storage`, so the SDK never mistakes an
  unreadable identity for "no identity" and silently generates a fresh one.

**Residual limits** (stated plainly): this raises the bar from "read a file" to
"defeat the platform key store", but does not protect a jailbroken device with
the screen unlocked, nor an in-process attacker while unlocked. Losing the device
is recoverable **without** exporting any secret — the account re-binds from a
fresh device by email verification (#6). See
[`docs/research/SECURE-STORAGE.md`](../../docs/research/SECURE-STORAGE.md).

On **Linux / headless / tests**, where `Security` doesn't exist,
`KeychainSecretStore` compiles out and `FileSecretStore` (a per-namespace,
plaintext, `0600` file store — **dev only**) is used so the package builds and
the test runs. It honors the same fail-closed and namespacing contract.

## Relationship to the rest of the repo

This project is **outside** the Cargo workspace on purpose — it depends on
`mycellium-sdk` only through the generated Swift binding + the native library,
never as a workspace member. The SwiftPM core builds/tests on Linux (and can run
in CI); the SwiftUI app is a Mac/Xcode acceptance step. See
[`docs/NATIVE-CLIENTS.md`](../../docs/NATIVE-CLIENTS.md).
